O que é um diagrama ER (DER)?

Primeiramente, o que é um Diagrama de Entidade-Relacionamento?

O Diagrama de Entidade-Relacionamento, também conhecido como DER, Diagrama ER ou modelo ER, é um tipo de diagrama estrutural usado no projeto de bancos de dados. Um DER contém diferentes símbolos e conectores que visualizam duas informações importantes: as principais entidades dentro do escopo do sistema e os relacionamentos entre essas entidades .

E é por isso que se chama diagrama de "Entidade" e "Relacionamento" (DER)!

Quando falamos de entidades em um DER (Diagrama Entidade-Relacionamento), geralmente estamos nos referindo a objetos de negócio como pessoas/funções (ex: Aluno), objetos de negócio tangíveis (ex: Produto), objetos de negócio intangíveis (ex: Registro), etc. "Relacionamento" diz respeito a como essas entidades se relacionam entre si dentro do sistema.

Em um projeto ER típico, você pode encontrar símbolos como retângulos arredondados e conectores (com diferentes estilos em suas extremidades) que representam as entidades, seus atributos e inter-relacionamentos.

Quando desenhar diagramas ER?

Então, quando devemos desenhar diagramas ER? Embora os modelos ER sejam desenvolvidos principalmente para o projeto de bancos de dados relacionais em termos de visualização de conceitos e projeto físico de bancos de dados, ainda existem outras situações em que os diagramas ER podem ser úteis. Aqui estão alguns casos de uso típicos.

  • Projeto de banco de dados - Dependendo da escala da mudança, alterar a estrutura de um banco de dados diretamente em um SGBD pode ser arriscado. Para evitar danos aos dados em um banco de dados de produção, é fundamental planejar as alterações cuidadosamente. O Diagrama Entidade-Relacionamento (DER) é uma ferramenta útil nesse processo. Ao desenhar diagramas ER para visualizar ideias de projeto de banco de dados, você tem a oportunidade de identificar erros e falhas de projeto, e de corrigi-los antes de implementar as alterações no banco de dados.
  • Depuração de banco de dados - Depurar problemas em um banco de dados pode ser um desafio, especialmente quando ele contém muitas tabelas, o que exige a escrita de SQL complexo para obter as informações necessárias. Ao visualizar o esquema do banco de dados com um diagrama ER, você obtém uma visão completa de todo o esquema. Você pode localizar facilmente entidades, visualizar seus atributos e identificar os relacionamentos que elas têm com outras. Tudo isso permite analisar um banco de dados existente e revelar problemas com mais facilidade.
  • Criação e atualização de banco de dados - O Visual Paradigm, uma ferramenta de diagrama ER, oferece suporte a uma ferramenta de geração de banco de dados que automatiza o processo de criação e atualização de bancos de dados por meio de diagramas ER. Assim, com essa ferramenta de diagrama ER, seu projeto ER deixa de ser apenas um diagrama estático e se torna um reflexo fiel da estrutura física do banco de dados.
  • Auxílio na coleta de requisitos - Determine os requisitos de um sistema de informação desenhando um diagrama ER conceitual que represente os objetos de negócio de alto nível do sistema. Esse modelo inicial também pode ser evoluído para um modelo físico de banco de dados que auxilie na criação de um banco de dados relacional ou na criação de mapas de processos e fluxogramas de dados.

Guia de notações ERD

Um diagrama ER contém entidades, atributos e relacionamentos. Nesta seção, vamos analisar detalhadamente os símbolos do DER.

Entidade

Uma entidade em um DER é uma coisa ou conceito definível dentro de um sistema , como uma pessoa/função (ex.: Aluno), um objeto (ex.: Fatura), um conceito (ex.: Perfil) ou um evento (ex.: Transação) (nota: em DER, o termo "entidade" é frequentemente usado em vez de "tabela", mas são sinônimos). Ao definir entidades, pense nelas como substantivos. Em modelos ER, uma entidade é representada por um retângulo com cantos arredondados, com seu nome na parte superior e seus atributos listados no corpo da forma da entidade. O exemplo de DER abaixo mostra um exemplo de uma entidade ER.

Atributos da Entidade

Também conhecido como coluna, um atributo é uma propriedade ou característica da entidade que o contém .

Um atributo possui um nome que descreve a propriedade e um tipo que descreve o tipo de atributo, como varchar para string e int para inteiro. Ao elaborar um diagrama ER para o desenvolvimento físico de um banco de dados, é importante garantir o uso de tipos suportados pelo SGBD relacional de destino.

Chave primária

Também conhecida como chave primária, uma chave primária é um tipo especial de atributo de entidade que define exclusivamente um registro em uma tabela de banco de dados . Em outras palavras, não pode haver dois (ou mais) registros que compartilhem o mesmo valor para o atributo de chave primária. O exemplo de diagrama ER abaixo mostra uma entidade 'Produto' com um atributo de chave primária 'ID' e uma pré-visualização dos registros da tabela no banco de dados. O terceiro registro é inválido porque o valor do ID 'PDT-0002' já está sendo usado por outro registro.

Chave estrangeira

Também conhecida como chave estrangeira (FK, na sigla em inglês), uma chave estrangeira é uma referência a uma chave primária em uma tabela . Ela é usada para identificar os relacionamentos entre entidades. Observe que as chaves estrangeiras não precisam ser únicas. Vários registros podem compartilhar os mesmos valores. O exemplo de diagrama ER abaixo mostra uma entidade com algumas colunas, entre as quais uma chave estrangeira é usada para referenciar outra entidade.

Relação

Uma relação entre duas entidades significa que elas estão associadas de alguma forma . Por exemplo, um aluno pode se matricular em um curso. A entidade Aluno está, portanto, relacionada ao Curso, e a relação é representada como um conector entre elas.

Cardinalidade

A cardinalidade define o número possível de ocorrências de uma entidade em relação ao número de ocorrências de outra . Por exemplo, UMA equipe tem MUITOS jogadores. Quando presentes em um diagrama ER, as entidades Equipe e Jogador estão interconectadas por meio de um relacionamento um-para-muitos.

Em um diagrama ER, a cardinalidade é representada por um pé de galinha nas extremidades dos conectores. Os três relacionamentos cardinais mais comuns são um-para-um, um-para-muitos e muitos-para-muitos.

Modelos de dados conceituais, lógicos e físicos

Um modelo ER é normalmente desenhado em até três níveis de abstração:

  • Diagrama ER conceitual / Modelo de dados conceitual
  • Diagrama ER lógico / Modelo de dados lógico
  • Diagrama ER físico / Modelo de dados físico

Embora todos os três níveis de um modelo ER contenham entidades com atributos e relacionamentos, eles diferem nos propósitos para os quais são criados e nos públicos-alvo que devem atingir.

De forma geral, os três modelos de dados são os seguintes: o analista de negócios utiliza um modelo conceitual e lógico para representar os objetos de negócio existentes no sistema, enquanto o projetista ou engenheiro de banco de dados elabora o modelo ER conceitual e lógico para produzir o modelo físico que apresenta a estrutura física do banco de dados, pronta para sua criação. A tabela abaixo mostra a diferença entre os três modelos de dados.

Modelo de dados conceitual

Os diagramas ER conceituais modelam os objetos de negócio que devem existir em um sistema e os relacionamentos entre eles . Um modelo conceitual é desenvolvido para apresentar uma visão geral do sistema, reconhecendo os objetos de negócio envolvidos. Ele define quais entidades existem, e NÃO quais tabelas. Por exemplo, tabelas com relacionamento "muitos para muitos" podem existir em um modelo de dados lógico ou físico, mas são representadas apenas como um relacionamento sem cardinalidade no modelo de dados conceitual.

Exemplo de modelo de dados conceitual

NOTA: O DER conceitual suporta o uso de generalização na modelagem da relação "um tipo de" entre duas entidades; por exemplo, Triângulo é um tipo de Forma. O uso é semelhante à generalização em UML. Observe que somente o DER conceitual suporta generalização.

Modelo de dados lógico

Um DER lógico é uma versão detalhada de um DER conceitual . Um modelo ER lógico é desenvolvido para enriquecer um modelo conceitual, definindo explicitamente as colunas em cada entidade e introduzindo entidades operacionais e transacionais. Embora um modelo de dados lógico seja independente do sistema de banco de dados no qual o banco de dados será criado, você ainda pode levá-lo em consideração se isso afetar o projeto.

Exemplo de modelo de dados lógico

Modelo de dados físicos

O diagrama ER físico representa o projeto concreto de um banco de dados relacional . Um modelo de dados físico detalha o modelo de dados lógico, atribuindo a cada coluna tipo, comprimento, nulidade, etc. Como um diagrama ER físico representa como os dados devem ser estruturados e relacionados em um SGBD específico, é importante considerar as convenções e restrições do sistema de banco de dados em que o banco de dados será criado. Certifique-se de que os tipos de coluna sejam suportados pelo SGBD e que palavras reservadas não sejam usadas na nomenclatura de entidades e colunas.

Como desenhar um diagrama ER?

Se você está com dificuldades para começar a desenhar um diagrama ER, não se preocupe. Nesta seção, daremos algumas dicas sobre diagramas ER. Siga os passos abaixo para entender como desenhar um diagrama ER de forma eficaz.

  1. Make sure you are clear about the purpose of drawing the ERD. Are you trying to present an overall system architecture that involves the definition of business objects? Or are you developing an ER model ready for database creation? You must be clear about the purpose to develop an ER diagram at the right level of detail (Read the section Conceptual, Logical and Physical Data Models for more details)
  2. Make sure you are clear about the scope to model. Knowing the modeling scope prevents you from including redundant entities and relationships in your design.
  3. Draw the major entities involved in the scope.
  4. Define the properties of entities by adding columns.
  5. Review the ERD carefully and check if the entities and columns are enough to store the data of the system. If not, consider adding additional entities and columns. Usually, you can identify some transactional, operational and event entities in this step.
  6. Consider the relationships between all entities and relate them with proper cardinality (e.g A one-to-many between entity Customer and Order). Don't worry if there are orphan entities. Although it's not common, it's legit.
  7. Apply the technique of database normalization to re-structure the entities in a way that can reduce data redundancy and improve data integrity. For example, the details of the manufacturer might be stored under the Product entity initially. During the process of normalization, you may find that the detail keeps repeating records over records, then you can split it as a separate entity Manufacturer, and with a foreign key that links between Product and Manufacturer.

Using ERD with Data Flow Diagram (DFD)

In system analysis and design, Data Flow Diagram (DFD) can be drawn to visualize the flow of information within system processes. In a Data Flow Diagram, there is a symbol called Data Store, which represents a database table that provides the information needed by the system.

Since a physical ER Diagram provides a blueprint of an actual database, the entities in such an ERD are aligned with datastores in a DFD. You can draw ERD as a complement to DFD by representing the structure of information that flows within a system, or, on the contrary, to draw DFD in complementing an ERD by showing how the data will be utilized by the system in runtime.

Using ERD with BPMN Business Process Diagram (BPD)

In business process mapping, BPMN Business Process Diagram (BPD) can be drawn to visualize business workflows. In a Business Process Diagram, there is a symbol called Data Object, which represents the data input into / output from process activities.

Como um modelo de dados conceitual e lógico fornece uma visão geral dos objetos de negócio dentro de um sistema, as entidades nesses diagramas ER estão alinhadas com os objetos de dados no diagrama de processos de negócio (BPD). Você pode desenhar um diagrama ER como complemento ao BPD, representando a estrutura dos objetos de dados necessários para um fluxo de trabalho de negócio ou, inversamente, desenhar um BPD como complemento a um diagrama ER, mostrando como os dados serão utilizados ao longo de um processo de negócio.
